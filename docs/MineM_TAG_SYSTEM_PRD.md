# MineM 标签体系 PRD

版本：P0 已确认，2026-07-18

## 1. 背景

现有 `assets.tags` 是逗号分隔的平铺文本。资产类型、媒体类型、导入动作、来源编号、版本、标题关键词和业务语义混合存放，导致同义标签、格式变体和无业务价值标签大量出现，无法支撑稳定筛选、AI 推荐和未来组织汇报池协作。

本需求以“先清空旧标签，再按新规则重新建立”为原则。清空标签不删除任何素材、预览、资源文件、编号、来源、版本或编排关系。

## 2. 目标

1. 标签只表达可检索的业务语义，不再承担类型、编号、版本和来源存储。
2. 提供统一词表、同义词归并和受控录入，避免标签继续膨胀。
3. 支持 AI 按页面内容、导入说明和素材属性生成候选标签，人工确认后生效。
4. 支持未来个人库、团队汇报池和组织资产库的共享与治理。
5. 保持现有导入、预览、搜索、版本、资源复用和汇报编排行为不变。

## 3. 非目标

- 本期不改变 `asset_type`、`media_kind`、`resource_kind`、`asset_code`、`source_type`、`upload_id`、版本字段或权限模型。
- 本期不把汇报标题、页面标题、来源路径自动变成标签。
- 本期不做复杂的组织审批流、跨组织权限或机器学习训练。

## 4. 新标签模型

### 4.1 结构化字段，不作为标签

以下内容继续使用现有字段展示、搜索和筛选，禁止写入业务标签：

- 素材类型：汇报、页面、资源、案例、故事线。
- 媒体与资源类型：HTML、图片、Logo、图标、GIF、视频、SVG。
- 编号与来源：`RPT`、`CTRL`、`RES` 编号、来源路径、导入批次、来源汇报。
- 生命周期：版本号、当前版本、系统合并、复用关系、自动同步、导入方式。
- 权限与状态：个人、团队、组织、待审核、已归档、已下架。该类为后续独立字段，不先伪装成标签。

### 4.2 受控业务标签维度

每个素材可在以下维度中选择标签；同一维度可多选，默认每个素材展示不超过 4 个标签。

| 维度 | 用途 | 第一版示例 |
| --- | --- | --- |
| 内容角色 | 页面或汇报表达什么 | 封面、目录、问题、价值、方案、案例、数据、流程、总结、行动 |
| 使用场景 | 在什么工作中复用 | 经营汇报、客户拜访、项目复盘、产品发布、内部培训、方案宣讲 |
| 目标对象 | 给谁使用或观看 | CEO、管理层、销售、客户、一线员工、技术团队 |
| 行业 | 适用行业 | 制造、零售、互联网、金融、医疗、教育、通用 |
| 业务主题 | 内容主题 | AI 转型、组织协同、效率提升、数字员工、客户增长、质量管理、风险管理 |

### 4.3 标签来源

- `manual`：人工在标签选择器中确认或移除。
- `ai_suggested`：AI 生成、尚未确认的候选标签，不进入正式筛选结果。
- `rule_seeded`：导入时由确定性规则给出的候选标签，不直接作为正式业务标签。
- `admin`：未来组织词表维护者创建或调整的标准标签。

第一版正式标签只保存人工确认结果；AI 和规则结果作为建议展示。

### 4.4 AI 理解证据优先级

AI 为每页生成候选标签时，必须记录证据来源、置信度与冲突状态，并按以下优先级判断：

| 优先级 | 证据 | 作用 |
| --- | --- | --- |
| P0 | 人工已确认标签或人工明确移除记录 | 最高优先级；AI 不得覆盖，仅可提示人工复核。 |
| P1 | 页面自身可读内容 | HTML 的标题、正文、表格、图表说明、`alt` 文本及页面截图的视觉识别结果；是内容角色和业务主题的主要依据。 |
| P2 | 汇报上下文 | 当前页页码、前后页内容、章节/编排位置、所属汇报主题；用于判断页面在叙事中的角色。 |
| P3 | 演讲稿、妙记、外部文档 | 按页面标题、页码或语义匹配到的段落；用于补足目标对象、表达意图和业务场景。 |
| P4 | 结构化来源信息 | 用户填写的导入说明、来源系统、案例组、资源分类；只能作为辅助证据。 |
| P5 | 文件名与路径 | 最弱提示；不得单独生成正式或高置信候选标签。 |

页面截图用于补足 Canvas、图片文字或视觉结构，不能仅凭品牌 Logo、背景图或装饰图判断业务主题。

### 4.5 冲突处理

1. 同一维度的 P0 与任何 AI 结论冲突时，保留人工结论，并记录“需复核”提示。
2. P1 与 P2/P3 冲突时，以页面自身内容为准；低优先级结论仅作为备选，不自动展示为推荐。
3. P1 内部的 HTML 文本与视觉识别发生冲突时：文本能明确表达主题则优先文本；页面主要由图片、Canvas 或截图承载信息时，提升视觉证据权重。
4. 同一维度出现多个互斥候选且最高分差距不足 `0.15`，或最高置信度低于 `0.70`，标记为“待确认”，不写入正式标签。
5. 不同维度允许并存，例如“案例 + 制造 + CEO + 经营汇报”不视为冲突；仅在同一维度的互斥值之间裁决。
6. AI 不得用来源编号、文件名、版本或资源类型填充业务标签；这些信息继续走结构化字段。

## 5. 核心交互

### 5.1 标签展示

- 卡片展示内容角色优先，其次场景、行业和主题；不展示技术或来源元数据。
- 详情页按维度分组展示全部正式标签，并显示“AI 推荐”区域。
- 无正式标签时显示“未分类”，不显示旧标签回退内容。

### 5.2 标签编辑

- 汇报、页面、资源和案例均可编辑业务标签。
- 编辑器按维度分区，使用可搜索的多选项；不提供自由文本直接落库。
- 用户可提交“建议新标签”，进入待规范化状态，不立即参与全局筛选。
- 删除标签仅影响当前素材，不影响同一版本的其他素材。

### 5.3 搜索与筛选

- 搜索编号、标题、来源路径时走结构化字段，不能依赖标签。
- 标签筛选支持多维 AND、同维 OR。例如：`制造` AND `CEO` AND (`方案` OR `案例`)。
- 同义词统一映射到标准标签。例如“数字化转型”“数智化升级”可归并到“AI 转型”或未来的“数字化转型”标准词。

### 5.4 AI 建议

- 标签分析与素材导入、资料同步彻底解耦。导入或同步成功后仅记录“待分析”状态，不在导入请求内生成候选标签，不阻塞入库、预览和页面加载。
- AI 只能从组织词表中选择；不能凭空创建正式标签。
- 置信度低、无法判断或跨维度冲突时不自动推荐。
- 候选结果必须返回证据摘要、证据优先级、置信度和冲突状态，供人工确认。

### 5.5 异步标签任务

标签生成采用独立异步任务，不随素材导入同步执行。

| 触发方式 | 适用对象 | 行为 |
| --- | --- | --- |
| 手动任务 | 单个素材、筛选结果、导入批次、指定汇报 | 用户明确选择范围后创建任务；可查看进度、失败原因和结果摘要。 |
| 定时任务 | 待分析或内容发生更新的素材 | 按配置周期扫描，不重复处理内容指纹未变化且已成功分析的素材。 |
| 重试任务 | 失败、超时或模型不可用的任务 | 保留失败原因，支持按单项或批次重试，并采用退避策略。 |

任务生命周期：`queued → running → completed / failed / cancelled`。任务取消只停止后续分析，不删除已保存的证据或人工确认结果。

执行边界：

1. 任务先收集页面自身内容、汇报上下文和可用外部文档证据，再调用 AI。
2. 任务产出仅为候选标签和证据，不直接写入正式标签。
3. 同一素材以内容指纹、分析模型版本和词表版本作为幂等键；命中相同组合时复用结果。
4. 页面截图或视觉分析属于可选的第二阶段任务；文本与结构提取先完成，避免拖慢常规任务。
5. 定时任务默认只处理“待分析、内容已变化、上次失败可重试”的素材；不对全库重复跑。

## 6. 旧标签清空与迁移

### 6.1 清空范围

对所有 `assets`、`asset_history` 与 `report_storyline_collections` 记录：

- 将 `tags` 清空为 `''`。
- 将 `assets.tag_seeded` 重置为 `0`，避免旧规则标签被误认为新体系数据。
- 不修改 `title`、`usage`、`snippet`、`asset_type`、`category`、`media_kind`、`resource_kind`、`asset_code`、`source_type`、`source_path`、`preview_url`、`upload_id`、版本字段、页面槽位、编排、故事线和资源引用。

### 6.2 执行保障

1. 执行前输出只读 dry-run、受影响表和记录数；项目运行目录不创建 SQLite 副本。
2. 清空操作必须提供默认只读预演和显式 `--apply`。
3. 操作后运行全量数据与链接测试，确保类型、编号、预览、页面槽位、版本父级、编排和资源引用均完整。
4. 不保留旧标签作为新标签回退数据；执行异常依赖单事务回滚，执行完成后不提供旧标签恢复入口。
5. P0 期间停止旧标签编辑和自动打标接口，导入及同步流程不得重新写入旧标签。

### 6.3 清空后的体验

- 现有列表不因标签为空而隐藏素材。
- 卡片仍可按类型、资源类型、编号、标题、来源路径和更新时间搜索。
- 原数据链路继续展示来源、批次、资源细分和版本信息。

## 7. 数据与接口演进

第一版沿用 `assets.tags` 存放已确认的标准业务标签，降低迁移风险；新增标签词表和建议表后再拆分为关系模型。

后续建议新增：

- `tag_definitions`：标准标签、所属维度、别名、状态、创建者、排序。
- `asset_tags`：资产与标准标签的多对多关系、来源、确认人、确认时间。
- `asset_tag_suggestions`：AI 或规则候选、置信度、状态和解释。

在关系表上线前，`assets.tags` 只允许写入标准业务标签的标准名。

## 8. 验收标准

1. 旧平铺标签被全部清空，维护脚本可重复验证清理结果。
2. 素材数量、主版本数量、汇报页面槽位、资源引用和故事线数量在清空前后保持一致。
3. 所有主素材详情、预览、真实汇报链接和编排接口均可访问。
4. 搜索编号、标题和链接仍可命中对应素材。
5. 新标签编辑器只写入受控业务标签；类型、编号、版本、来源不得再次作为业务标签写入。
6. AI 建议不阻塞导入和列表首屏加载。
7. 同一维度的冲突候选不会自动写入正式标签；人工结论不会被 AI 覆盖。
8. 导入完成不等待标签生成；手动和定时标签任务均可观察状态、取消和重试，且不会重复分析未变化的素材。
