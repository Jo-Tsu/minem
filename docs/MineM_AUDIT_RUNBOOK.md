# MineM 走查执行规范

本文档把 MineM 平台走查拆分为两套独立流程。每次开始前先根据用户指令选择流程，不默认扩大检查范围。

## 流程选择

| 用户指令 | 执行流程 |
| --- | --- |
| 数据走查、数据检查、素材数据治理、编号检查、重复素材检查 | 流程一：数据走查 |
| 检查预览、预览有问题、尺寸不对、图片空白、页面显示异常 | 流程二：预览走查 |

如果预览问题最终由脏数据或错误路径引起，先报告跨层根因，再输出引用清单和 dry-run，随后以单事务执行最小范围的数据修复。如果数据走查发现预览异常，只记录为待执行的预览问题，不顺带大范围修改前端样式。

## 流程一：数据走查

### 目标

保证素材分类、编号、来源、版本、引用、排序和文件关系正确，不产生垃圾数据、重复资产、断链资源或错误汇报。

### 执行步骤

1. 明确本轮涉及的模块和数据范围。
2. 检查 Docker、本地服务、端口和数据库实例，保证同一数据库只有一个服务实例写入。
3. 记录素材数量、接口状态、构建状态和数据基线。
4. 修复前输出 dry-run、引用清单和预计变更记录；项目运行目录不创建数据库副本。
5. 对照 PRD 检查素材类型与编号：
   - `RPT-*` 只能是完整汇报素材。
   - `CTRL-*` 只能是单页页面素材。
   - `RES-*` 只能是资源素材。
   - `CASE-*` 是案例页面组，不属于汇报素材。
   - 无汇报 Manifest 的多页面资源包逐页生成 `CTRL-PAGE-*`，不得推断生成 `RPT`。
6. 检查数据库记录、`upload_id`、`source_path`、`preview_url` 和实际文件是否一致。
7. 检查导入链路：HTML、ZIP、资源文件、完整汇报、自动扫描和案例生成分别处理。
8. 检查页面是否正确进入页面素材库，资源是否正确进入资源素材库。
9. 检查重复内容和版本组：相同内容优先复用，重复项合并为版本，不删除仍被引用的文件。
10. 检查引用关系、孤立版本、失效页槽、候选页和故事线来源。
11. 检查排序：
    - 汇报按新增或更新时间倒序。
    - 页面按新增、更新、版本新增或汇报引用时间倒序。
    - 资源按新增、更新、复用或版本新增时间倒序。
    - 排序必须在后端分页前完成。
12. 检查后端事务、错误处理、路径安全、并发写入和重复查询。
13. 检查前端列表、筛选、分页、删除、重命名、合并和任务状态是否与接口一致。
14. 按“数据关系、文件路径、导入机制、接口、前端展示”的顺序分批修复。
15. 重建并重启 Docker，重新执行全量数据审计。

### 验收标准

- 编号与类型错误为 `0`。
- 数据库文件缺失为 `0`。
- HTML 本地引用错误为 `0`。
- 重复资产按版本规则处理，不出现空白主版本。
- 页面和资源跨页排序正确。
- API 契约、Python 编译、TypeScript 检查和生产构建通过。
- Docker 服务正常且不存在重复实例。
- 治理脚本重复执行不产生新的数据变更。

### 输出内容

- 检查范围与数据基线。
- 问题清单、严重程度和根因。
- 数据变更数量与修复清单。
- dry-run 报告与事务执行结果。
- 验证结果与剩余风险。

## 流程二：预览走查

### 目标

保证素材在列表卡片、详情窗口和完整页面中的预览内容正确、尺寸稳定、资源完整，并且修复不显著影响加载速度。

### 执行步骤

1. 明确目标素材、问题页面和出现问题的预览形态。
2. 记录修复前 URL、HTTP 状态、窗口尺寸、页面尺寸和截图。
3. 检查原始页面设计尺寸、Manifest 尺寸、数据库预览元数据是否一致。
4. 区分 `16:9` 页面、非标准比例页面和长页面，禁止统一强制套用 `1920×1080`。
5. 检查缩略图来源、生成时间、缓存指纹和页面更新时间，确认是否使用旧缩略图。
6. 检查列表卡片：
   - 卡片尺寸稳定。
   - 内容等比缩放并居中。
   - 不拉伸、不裁切关键内容、不溢出。
   - 懒加载不能导致卡片高度跳动。
7. 检查详情弹窗或抽屉：
   - 预览区域不超出窗口。
   - 操作按钮、标题和预览不互相遮挡。
   - iframe 根据容器重新计算比例。
8. 检查完整汇报、临时汇报和案例页面组：
   - 上一页、下一页、页码和全屏正常。
   - 每次切页重新计算页面尺寸。
   - 页面更新后刷新对应预览和缩略图。
9. 检查图片、视频、SVG、字体、CSS 和脚本请求，确保没有 `404`、空白或跨目录断链。
10. 检查代表性窗口：`1920×1080`、`1440×900`、`1280×800` 和约 `768px` 宽的应用窗口。
11. 检查 DOM 尺寸：页面和预览容器不产生非预期横向溢出。
12. 检查浏览器控制台错误和网络错误。
13. 优先修复尺寸元数据、容器约束和缩放公式；只刷新被修改素材的缩略图。
14. 不增加列表实时 iframe 数量，不让完整页面在列表中提前加载。
15. 修复后重新截图，对比桌面和窄窗口，并检查加载时间和请求数量。

### 验收标准

- 目标 URL 返回 `200`，旧入口按产品规则跳转或失效。
- 图片、视频和 iframe 断链数量为 `0`。
- 预览宽高比与源页面一致。
- 页面不出现非预期横向滚动。
- 列表卡片、详情窗口和完整预览均无明显裁切、拉伸和遮挡。
- 翻页后尺寸不跳动，页码和全屏正常。
- 页面更新后预览能够刷新，不继续使用旧缓存。
- 修复不增加首屏预览请求，不显著降低列表加载速度。

### 输出内容

- 问题 URL、素材编号和预览形态。
- 修复前后的尺寸、截图和资源状态。
- 根因属于源页面、缩略图、容器、缩放还是缓存。
- 修改文件和影响范围。
- 桌面、窄窗口、控制台和加载性能验证结果。

## 协作约束

- 其他协作者或后台系统可能同时新增页面，不把无关新页面视为错误。
- 不删除、覆盖或重新编号本轮范围之外的页面。
- 只验证和修改本轮目标页面、入口、资源和 Manifest。
- 数据修复必须先输出 dry-run 并使用事务；预览修复必须保留可回退入口。
